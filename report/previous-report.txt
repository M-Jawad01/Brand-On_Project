Introduction
Problem Statement
Local banner printing shops operate entirely offline. Customers must visit the shop in person to browse material options, negotiate prices, and place orders. There is no way for customers to see available materials, compare prices, or track order status remotely. This creates inconvenience for customers and limits the shop's reach.
Proposed Solution
BrandON is a web-based platform that digitalizes the banner ordering workflow. It allows customers to:
    • Browse available banner materials (Star Flex, Vinyl, PVC, Backlit, etc.) with transparent per-square-foot pricing
    • Specify custom dimensions (width × height in feet) and quantity
    • Upload a design file (image/PDF)
    • Place an order online with automatic price calculation
    • Receive order confirmation via email
The shop admin can:
    • Add, edit, and manage banner materials and pricing
    • View and process incoming orders (Pending → Confirmed → In Progress → Completed)
    • Manage a portfolio/gallery of past work
    • Update company profile information

The platform is designed for a single banner printing shop. Multi-store or franchise support is explicitly out of scope. Payment integration is not included — the shop operates on Cash on Delivery (COD). User registration is not required for customers; orders are placed with just a name and phone number.

Sequence diagram:


                               Figure: Place Order Sequence Diagram

Description: This diagram illustrates the step-by-step process and interactions required to successfully place an order within the system.
    • User Interaction: The process begins when the user fills out the order form and submits it through the UI.
    • API Request: A POST request is sent to the order API, triggering the backend workflow.
    • Validation & Calculation: The server validates the input data and calculates the total price based on the selected material and dimensions.
    • Database Operation: Using the Prisma ORM, the system creates a new record in the database and generates a unique order number.
    • Confirmation: Once the data is saved, the system returns the order number to the user and displays an "Order Placed" confirmation message.

Design Pattern Descriptions and Their Purposes
In this project, we have successfully implemented two crucial design patterns to ensure our codebase is efficient, scalable, and maintainable.
1. Singleton Design Pattern
    • Purpose: To manage the database connection efficiently and prevent resource exhaustion.
    • Implementation and Usage: We implemented the Singleton pattern for our Database Client. The Database Client class is designed with a private constructor, which strictly prevents any external module from instantiating it directly using the new keyword. Instead, we use a static method called getInstance(). When getInstance() is called, it checks if an instance of the database client already exists. If it does, it returns that exact same instance; if not, it creates it first. Because of this approach, no matter how many files or modules import this database module, they all share the exact same database connection.
    • Benefit: This approach significantly saves system resources and prevents the database from running out of available connections by avoiding redundant connection creations.

2. Strategy Design Pattern
    • Purpose: To handle dynamic pricing calculations based on different customer needs and order conditions.
    • Implementation and Usage: We utilized the Strategy pattern specifically in our pricing section. We defined a PricingStrategy interface containing a calculate method. Based on this interface, we implemented three distinct concrete strategies:
        1. Standard Pricing: Performs a standard calculation by multiplying the single area by the quantity.
        2. Bulk Discount Pricing: Applies a 10% discount on the total price when the order quantity is 5 or more.
        3. Rush Order Pricing: Adds a 25% surcharge to the total price for urgent orders.
    • Our PriceCalculator class acts as the Context. It accepts the chosen strategy and delegates the calculation task to it. The strategy can be seamlessly changed at runtime using a setStrategy method.
    • Benefit: This implementation perfectly follows the Open/Closed Principle. It allows us to introduce new pricing rules in the future without modifying the existing, tested code.
